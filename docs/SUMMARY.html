Part 1

Theoretical Background

Understanding the core concepts behind this assignment: what coding agents are, how they are benchmarked, and why MLOps discipline matters for experiment reproducibility.

🤖 What Are Coding Agents?

Coding agents (e.g., Claude Code, Codex, Cursor, mini-swe-agent) are LLM-powered systems that autonomously write code to solve programming tasks. They operate in an agent loop: receive a task → think → execute a command → observe the output → repeat until the task is solved. The agent produces a patch (a git diff) as its final output.

📊 SWE-bench: The Benchmark

SWE-bench is a benchmark from Princeton NLP consisting of real GitHub issues and their verified test suites. The agent receives an issue description, works inside a Docker container with the repository at the exact commit, and produces a patch. SWE-bench then applies the patch and runs the original unit tests. If the previously-failing tests now pass, the instance is "resolved".

🧪 Why Evaluation Pipelines Matter

Without an automated pipeline, researchers SSH into a VM, manually run scripts, wait for results, and paste numbers into a spreadsheet. This leads to: no run history, no reproducibility, no queue for concurrent experiments, and no reliable way to answer "which config produced this result?" An MLOps pipeline solves all of these problems by providing durability, repeatability, provenance, and operational visibility.

🔬 mini-swe-agent

mini-swe-agent is a lightweight coding agent framework from the SWE-agent team. It uses an LLM (e.g., Kimi-K2.6 from Moonshot AI, hosted on Nebius) to interact with a bash shell inside a Docker container. The agent reads files, edits code, runs tests, and ultimately submits a trajectory (full conversation log) and a prediction (the git patch).

Docker-in-Docker (DinD) Pattern

The DockerOperator spawns ephemeral containers for heavy workloads (agent + evaluator). These containers mount the host's /var/run/docker.sock because mini-swe-agent itself needs Docker to create the SWE-bench test environments. This is the "Docker-in-Docker" pattern — the Airflow container spawns another container, which in turn spawns yet another container for the coding task.

Stage 1: prepare_run

Python @task
Reads Airflow UI parameters (split, subset, workers, model, task_slice, run_id, cost_limit), generates a unique run ID if none is provided, creates the runs/<run-id>/ directory tree, and writes config.json. The config dict is passed to downstream tasks via XCom.

Stage 2: run_agent

DockerOperator
Spawns an ephemeral mlops-pipeline:latest container that runs uv run mini-extra swebench with Jinja-templated parameters pulled from XCom. The agent interacts with an LLM (Kimi-K2.6) via the Nebius API, spawns a SWE-bench Docker container for the coding task, produces a trajectory and a prediction patch, and writes everything to runs/<run-id>/run-agent/.

Stage 3: run_eval

DockerOperator
Another ephemeral container runs the SWE-bench evaluation harness against the agent's preds.json. SWE-bench applies the patch inside a fresh container, runs the project's test suite, and records pass/fail results. A post-eval script copies logs and per-instance report.json files into runs/<run-id>/run-eval/.

Stage 4: summarize_and_log

Python @task
Parses evaluation results, aggregates metrics (total, resolved, unresolved, resolve_rate), writes metrics.json and manifest.json, logs all parameters, metrics, and artifacts to MLflow, and auto-updates the REPORT.md "Concrete Run Example" section with real data.

Pipeline Helper Functions

📦 build_run_config()

Reads Airflow params, auto-generates run_id if empty, resolves dataset_name from subset, and returns a complete config dict.

📁 prepare_run_dir()

Creates the runs/<run-id>/ directory tree with run-agent/ and run-eval/ subdirectories. Raises FileExistsError to prevent artifact collision.

📊 collect_metrics()

Parses evaluation results from results.json. If the top-level summary is empty, it falls back to aggregating per-instance report.json files from the reports/ directory. Handles Docker permission issues by deleting root-owned files.

📋 build_manifest()

Creates manifest.json — an index file pointing to all key artifacts (config, predictions, trajectories, eval logs, metrics). Enables single-folder reproducibility.

📈 log_mlflow_run()

Logs run parameters, all metrics, and key artifact files (config.json, metrics.json, preds.json) to MLflow for cross-run comparison.

📝 update_report_run_example()

Auto-rewrites REPORT.md Section 4 with real run config and metrics. Ensures the report never drifts from committed artifacts.

Part 7

Challenges & Solutions

Real engineering problems encountered during implementation and how they were resolved.

🐳 Docker Container Name Conflict

Problem: SWE-bench names evaluation containers using the run_id. Reusing a run_id caused a 409 Conflict error because the old container still existed.

Solution: Always use a unique run_id. The pipeline auto-generates timestamp-based IDs when none is provided, and prepare_run_dir() raises FileExistsError to prevent collisions.

🔒 Root-Owned File Permissions

Problem: Docker containers run as root, creating files owned by root. When Airflow (running as UID 50000) tried to overwrite results.json, it got PermissionError.

Solution: Use os.remove() first (directory ownership allows deletion), then create a fresh file. Fallback catches ensure the pipeline never crashes.

📄 Empty results.json

Problem: SWE-bench writes its summary file with a model-specific name (e.g., nebius__moonshotai__Kimi-K2.6.test.json). The copy command sometimes fails, leaving results.json as {}.

Solution: Added a bulletproof fallback in collect_metrics() that aggregates per-instance report.json files from the reports/ directory when the top-level summary is missing.

📝 Report Drift

Problem: REPORT.md was written manually and became inconsistent with the actual committed run artifacts.

Solution: Added update_report_run_example() to pipeline.py that auto-rewrites Section 4 of REPORT.md from real run_config and metrics. Called automatically by the DAG's summarize_and_log task.

Part 9

How to Reproduce

Step-by-step instructions to run the entire pipeline from scratch on a fresh VM.

1. Provision a VM

CPU VM with 8 CPU, 32 GB RAM, public IP on Nebius. Install Docker, Docker Compose, and uv.

2. Clone & Configure

git clone <repo> && cd mlops-assignment-e2e-ml-pipeline
cp .env.example .env — add your NEBIUS_API_KEY

3. Build & Launch

docker build -t mlops-pipeline:latest .
mkdir -p runs logs && chmod -R 777 runs logs
sudo chmod 666 /var/run/docker.sock
docker compose up -d

4. Trigger a Run

Open Airflow at http://localhost:8080 (admin/admin). Find evaluate-agent-docker, click Play → Trigger DAG w/ config. Set your desired parameters and click Trigger.
