Step 1 Prerequisites

What you need before starting.

RequirementDetails
VMCPU VM with 8 vCPU, 32 GB RAM, public IP (Nebius Cloud recommended)
OSUbuntu 22.04 or 24.04 LTS
SSH KeyYour public SSH key added to the VM
Nebius API KeyFrom Nebius Studio — needed for LLM inference
Local MachineSSH client, web browser for Airflow UI (port 8080) and MLflow UI (port 5000)
💡 No GPU required

The agent uses a hosted LLM API (Kimi-K2.6 on Nebius). All compute is CPU-based for Docker containers.

SSH Config (Recommended)

Add your VM to ~/.ssh/config for easier access:

~/.ssh/config
Host mlops-vm
  HostName <YOUR-VM-PUBLIC-IP>
  User <YOUR-USERNAME>
  ForwardAgent yes
  IdentityFile ~/.ssh/id_rsa

Then connect with: ssh mlops-vm

Step 2 VM & Docker Installation

Install Docker, Docker Compose, and uv on your fresh VM.

2.1 — Install uv (Python package manager)

bash
curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc   # reload PATH

2.2 — Install Docker Engine

bash
# Install prerequisites
sudo apt update
sudo apt install -y ca-certificates curl

# Add Docker's official GPG key
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Add Docker repository
sudo tee /etc/apt/sources.list.d/docker.sources <<EOF
Types: deb
URIs: https://download.docker.com/linux/ubuntu
Suites: $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}")
Components: stable
Architectures: $(dpkg --print-architecture)
Signed-By: /etc/apt/keyrings/docker.asc
EOF

sudo apt update

# Install Docker + Compose plugin
sudo apt install -y docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin

2.3 — Allow Docker without sudo

bash
sudo groupadd -f docker
sudo usermod -aG docker "$USER"
sudo chmod 666 /var/run/docker.sock

# Verify
docker run hello-world

2.4 — Verify installations

bash
docker --version        # Docker version 27.x+
docker compose version  # Docker Compose v2.x+
uv --version            # uv 0.x+

Step 3 Project Setup

Clone the repo, configure environment variables, and build the Docker image.

3.1 — Clone the repository

bash
git clone <YOUR-REPO-URL>
cd mlops-assignment-e2e-ml-pipeline

3.2 — Configure environment variables

bash
cp .env.example .env
nano .env

Edit .env and set your Nebius API key:

.env
# ── Inference ──
NEBIUS_API_KEY=your_actual_nebius_api_key_here

# ── MLflow tracking ──
MLFLOW_TRACKING_URI=http://mlflow:5000

# ── S3 / Object Storage (optional) ──
S3_ENDPOINT_URL=
S3_BUCKET=mlops-assignment-artifacts
AWS_ACCESS_KEY_ID=
AWS_SECRET_ACCESS_KEY=
⚠️ Critical: Set MLFLOW_TRACKING_URI correctly

For Docker Compose mode, use http://mlflow:5000 (Docker DNS).
For standalone mode, use http://localhost:5000.

3.3 — Install Python dependencies

bash
uv sync
source .venv/bin/activate

3.4 — Build the Docker image

This image is used by Airflow's DockerOperator for the agent and evaluation tasks:

bash
uv lock
docker build -t mlops-pipeline:latest .

This builds an Ubuntu 24.04 container with Docker CLI, uv, and all project Python dependencies locked via uv.lock.

💡 Build time

First build may take 3-5 minutes due to dependency installation. Subsequent builds use Docker cache.

3.5 — Set directory permissions

Airflow runs as a non-root user (UID 50000). Docker containers create files as root. We need broad permissions to prevent PermissionError:

bash
mkdir -p runs logs
chmod -R 777 runs logs

3.6 — (Optional) Clone reference repositories

Not required for running, but useful for debugging agent behavior:

bash
cd ..
git clone https://github.com/SWE-agent/mini-swe-agent.git
git clone https://github.com/swe-bench/SWE-bench.git
cd mlops-assignment-e2e-ml-pipeline

Step 4 Deploy with Docker Compose

Launch the full infrastructure: PostgreSQL, Airflow (webserver + scheduler), and MLflow.

4.1 — Start all services

bash
docker compose up -d

This starts 5 containers:

ServicePortPurpose
postgres5432Airflow metadata database
airflow-init—One-shot: DB migration + admin user creation
airflow-webserver8080Airflow web UI
airflow-scheduler—Executes DAG tasks via LocalExecutor
mlflow5000MLflow tracking server with SQLite backend

4.2 — Wait for initialization

Airflow needs ~60 seconds for database migration. Monitor progress:

bash
# Watch all containers
docker compose logs -f

# Check health specifically
docker compose ps

You should see all services as Up (healthy) or Up.

4.3 — Set up SSH port forwarding

From your local machine, forward ports to access the web UIs:

local terminal
ssh -L 8080:localhost:8080 -L 5000:localhost:5000 mlops-vm

Then open in your browser:

ServiceURLCredentials
Airflowhttp://localhost:8080admin / admin
MLflowhttp://localhost:5000No auth required

4.4 — Verify Airflow sees the DAG

  1. Open http://localhost:8080
  2. Login with admin / admin
  3. You should see evaluate-agent-docker in the DAG list
  4. If not, wait 30 more seconds — the scheduler needs to parse the DAG file

Alternative: Standalone Mode (No Docker Compose)

For quick testing without Docker Compose:

bash
# Terminal 1: Start Airflow
bash run-airflow-standalone.sh

# Terminal 2: Start MLflow
uv run mlflow server --host 0.0.0.0 --port 5000
💡 Standalone vs Docker Compose

Standalone mode uses SQLite for Airflow and runs directly on the host. Docker Compose mode uses PostgreSQL and is the production-recommended approach.

Step 5 Run the Evaluation Pipeline

Trigger a full agent → evaluation → metrics → MLflow run from the Airflow UI.

5.1 — Trigger the DAG

  1. Open Airflow UI
  2. Find the evaluate-agent-docker DAG
  3. Click the ▶ Play button → Trigger DAG w/ config

5.2 — Configure parameters

Fill in the configuration form:

ParameterRecommended ValueDescription
splittestDataset split
subsetverifiedSWE-bench Verified (500 tasks)
workers5Parallel agents (1-20)
modelnebius/moonshotai/Kimi-K2.6LLM model ID on Nebius
task_slice0:1Run only 1 task (quick test)
run_idmy_run_001Unique identifier (auto-generated if empty)
cost_limit2.0Max USD per agent instance
⚠️ Important: Use a unique run_id every time

SWE-bench creates Docker containers named after the run_id. Reusing an ID causes a 409 Conflict error. Either enter a unique name or leave it blank for auto-generation.

5.3 — Monitor execution

Click on the running DAG to see the 4-stage pipeline progress:

StageExpected DurationWhat It Does
prepare_run~1 secondCreates run directory + config.json
run_agent~2-5 minutesLLM agent solves the coding task inside Docker
run_eval~1-2 minutesSWE-bench evaluates the agent's patch
summarize_and_log~2 secondsParses results, writes metrics, logs to MLflow

Click on any task → Log to see real-time output.

5.4 — Verify the run completed

On the server, check the generated artifacts:

bash
# List the run directory
ls -la runs/my_run_001/

# Check metrics
cat runs/my_run_001/metrics.json

# Check results
cat runs/my_run_001/run-eval/results.json

# Check manifest
cat runs/my_run_001/manifest.json

Expected metrics.json output:

metrics.json
{
  "total_instances": 1,
  "submitted_instances": 1,
  "completed_instances": 1,
  "resolved_instances": 1,
  "unresolved_instances": 0,
  "error_instances": 0,
  "resolve_rate": 1.0
}

Step 6 Download Results to Local Machine

Copy the run folder from the remote server for committing to your repository.

6.1 — Download via SCP

From your local machine:

local terminal
# Download the entire run folder
scp -r mlops-vm:~/mlops-assignment-e2e-ml-pipeline/runs/my_run_001 \
  ./runs/my_run_001

6.2 — Commit the run to your repo

local terminal
# Update .gitignore to track this specific run
# (edit the !/runs/XXXX/ line to match your run_id)

git add runs/my_run_001 .gitignore REPORT.md
git commit -m "docs: add authentic evaluation run my_run_001"
git push

6.3 — Verify the folder structure

Your committed run should contain:

expected structure
runs/my_run_001/
├── config.json                    # Run parameters
├── metrics.json                   # Aggregated metrics
├── manifest.json                  # Index of all artifacts
├── run-agent/
│   ├── preds.json                 # Agent's predictions (patches)
│   ├── minisweagent.log           # Agent execution log
│   └── <instance_id>/
│       └── <instance_id>.traj.json  # Full agent trajectory
└── run-eval/
    ├── results.json               # Aggregated evaluation summary
    ├── reports/                    # Per-instance pass/fail reports
    └── logs/                      # SWE-bench container logs

Step 7 View Results in MLflow

Compare runs, analyze metrics, and download artifacts from the MLflow UI.

7.1 — Open MLflow

Navigate to http://localhost:5000 (with port forwarding active).

7.2 — Find your experiment

  1. Select the swe-bench-evaluation experiment from the sidebar
  2. Each pipeline execution appears as a separate MLflow Run

7.3 — Explore a run

Click on a run to see:

  • Parameters: model, split, subset, workers, run_id, task_slice, cost_limit
  • Metrics: total_instances, resolved_instances, resolve_rate, etc.
  • Artifacts: config.json, metrics.json, manifest.json, preds.json

7.4 — Compare multiple runs

  1. Select multiple runs using the checkboxes
  2. Click Compare
  3. View side-by-side parameter and metric comparisons

Step 8 Troubleshooting

Common issues and their solutions.

🐳 409 Conflict: Container name already in use

Cause: You reused a run_id from a previous run. SWE-bench names containers after the run_id.

fix
# Remove the stale container
docker rm -f sweb.eval.astropy__astropy-12907.<old_run_id>

# Or just use a new, unique run_id next time

🔒 PermissionError: Permission denied on results.json

Cause: Docker containers create files as root. Airflow runs as UID 50000.

fix
# Reset permissions on the runs directory
sudo chmod -R 777 runs/

# The pipeline code already handles this gracefully via os.remove() fallback

🌐 DAG not showing in Airflow UI

Cause: Scheduler hasn't parsed the DAG file yet, or there's a Python import error.

fix
# Check for import errors
docker compose logs airflow-scheduler | grep -i error

# Force re-parse by touching the DAG file
touch dags/evaluate_agent_docker.py

# Wait 30 seconds, then refresh the Airflow UI

🔑 Agent fails with API key error

Cause: NEBIUS_API_KEY not set or not propagated to Docker containers.

fix
# Verify the key is set in your .env file
grep NEBIUS_API_KEY .env

# Restart services to pick up .env changes
docker compose down
docker compose up -d

📦 Docker image not found: mlops-pipeline:latest

Cause: You forgot to build the Docker image, or built it in a different context.

fix
# Build the image
docker build -t mlops-pipeline:latest .

# Verify it exists
docker images | grep mlops-pipeline

🔄 MLflow connection refused

Cause: MLflow container not running, or MLFLOW_TRACKING_URI misconfigured.

fix
# Check if MLflow is running
docker compose ps mlflow

# For Docker Compose, use http://mlflow:5000 (not localhost)
# For standalone mode, use http://localhost:5000

# Restart if needed
docker compose restart mlflow

⏱️ Agent task times out or hangs

Cause: The agent is waiting for LLM responses. Each task can take 2-5 minutes.

fix
# Check live logs of the running agent container
docker ps  # find the mlops-pipeline container
docker logs -f <container_id>

# If truly stuck, the DAG has execution_timeout=None (no timeout)
# You can manually kill the container:
docker kill <container_id>

Useful Debug Commands

bash
# View all running containers
docker ps

# View all Airflow logs
docker compose logs -f airflow-scheduler

# Check disk space (Docker images can be large)
df -h

# Clean up old Docker containers and images
docker system prune -f

# Restart everything from scratch
docker compose down -v
docker compose up -d
